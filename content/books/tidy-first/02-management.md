# 파트2 관리

## 16장 코드 정리 구분

코드 정리와 동작 변경을 섞지 말 것

- 변경 할 때 의도적으로 쉽게 변경하던가 변경하기 쉽게 만들던가, 둘 중 하나는 목표로 삼는 것이 좋다.
    
- 변경 사항을 나누어 별도의 PR로 만들어야 한다. 순서가 있는 일련의 코드 정리는 PR 한 개로 만든다.
    
- **동작 변경이 있으면 별도의 피알을 만들고, 코드 정리와 동작 변경사이를 번갈아 가면서 전환할 대마다 새 PR을 만들어야한다.**
    
    > 이러다가 하루종일 PR만 만들겠어..
    
- 크고 포괄적인 PR로 만들면 전체 그림을 보여줄수 있다. 하지만, 검토자에게는 유용한 피드백을 제공하기에는 너무 큰 덩어리일 수 있다.
    
- 아주 작은 PR은 신속한 검토 및 소소한 피드백을 유도할 수 있다. 하지만, 무시될 우려 존재한다.
    
    > 왜 무시될까? 빠르고 신속하게 진행될텐데. 가볍게 보여서 그런가
    > 
    > - 너무 많으면 리뷰어 입장에서 **알림 피로**가 생김
    > - 작아 보여서 **긴급하지 않다**고 느껴짐
    > - “이건 나중에 봐도 되겠지” 하다가 밀림
    > - 팀 문화상 작은 PR을 빠르게 처리하는 습관이 없으면 오히려 쌓임
    
- 대체로 코드를 느리게 검토한다면, 더 큰 PR을 만드는 결과를 초래하여 향후 검토를 더욱 느리게 만든다.
    
- 코드 정리에 익숙해지고, 작은 단위로 작업하고, 안전하게 작업하는데 익숙해지면 코드 정리 PR을 따로 검토하지 않는 것을 목표할 것 ⇒ 대기 시간 감소 및 더 작게 PR을 만들도록 장려하는 효과 발생
    
    > 코드 정리에 국한된 말이겠지
    > 
    > - 팀이 코드 정리 패턴에 익숙하고
    > - 변경이 작고
    > - 동작 변경이 없고
    > - 테스트/관례가 잘 갖춰져 있다면
    

## 17장 연쇄적인 정리

- 코드 정리는 자꾸 손이 가는 감자칩과 같다.
    
    > 작가는 감자칩을 좋아하나부다.
    
- 코드를 계속 정리하고 싶은 충동을 관리하는 것도 코드 정리의 핵심 기술이다.
    
- 아주 작은 단계로 나누어 코드를 정리해볼 것
    

어떤 정리를 할 것인지

- **보호 구문**
    
    - 조건이 명시적으로 보여져 도움을 줄 수 있다.
    - 중첩을 줄여서 흐름과 변경지점을 드러내게 함
- **안 쓰는 코드**
    
    - 코드 읽는 순서에 맞춰 정렬, 응집도 높이는 배치가 보인다.
    - 인지 부하를 줄이는 정리
- **대칭으로 맞추기**
    
    - 유사한 코드들이 묶여진 순서대로 읽을 수 있다. 비슷한코드를 모아 상단에 배치해서 목차 같이 사용.
- **새로운 인터페이스로 기존 루틴 부르기**
    
    - 새로운 인터페이스를 먼저 만들고 내부에서는 기존 구현을 호출하게 해서 호출부를 천천히 새 인터페이스로 옮길 수 있게 만드는 것
        
    - 인터페이스 변경하고 나서는 자동 변환 도구가 없으면 직접 일일이 변환해줘야한다.
        
    - 팬아웃 현상
        
        > 팬아웃??? - 이 변경의 영향이 퍼져나가는 범위
        
- **읽는 순서**
    
    - 읽는 순서 정리하면 대칭을 맞추거나 응집도를 올릴 수도 있다.
- **응집도를 높이는 배치**
    
    - 함께 이해해야 하고 함께 바뀌는 것들을 가까이 두라
- **설명하는 변수**
    
    - 설명하는 변수 = 설명하는 도우미 후보
- **설명하는 상수**
    
    - 한 번에 바뀌는 상수를 모아서 묶어놓으면 바꾸기 좋지
- **명시적인 매개변수**
    
- **비슷한 코드끼리**
    
- **도우미 추출**
    
- **하나의 더미**
    
- **설명하는 주석**
    
- **불필요한 주석 지우기**
    
    > 사실 작가는 주석 반대론자
    

코드 정리로 PR쓰기엔 뭔가…머쓱하네요

## 18장 코드 정리의 일괄 처리량

코드 정리를 한꺼번에 할 것인지, 모두 개별적으로 할 것인지, 그 중간을 택할건인지 범위 판단 필요

- 골디락스 딜레마로 알려진 타협점이 필요한 상황
    
    - 골디락스라는 어린 소녀가 세 그릇의 죽을 맛보면서 너무 뜨겁지도 너무 차갑지도 않고 온도가 딱 맞는 죽을 좋아한다는 이야기
    
    > 아 딱 적정선의 타협점이 필요하다. 라는 말
    
- 일괄 처리당 코드 정리수가 많을 수록 ⇒ 충돌, 상호작용, 추측이 증가
    
    - **충돌** : 통합 과정 지연, 다른 사람 작업과 충돌 발생, 병합 시간 증가
        
    - **상호작용** : 우연히 다른 동작도 변경할 수 있음
        
    - **추측** : 처리하는 코드 정리수가 많을 수록 자연스럽게 더 많은 코드 정리함
        
        > 왜 추측으로 이름을 지었는가.. - 투기적 정리, 필요하지 않을지도 모르는 정리까지 하게 되는 것
        
- 하나의 변경 사항을 검토하고 배포하는 데 드는 고정 비용은 상당히 많다.
    
    > 코드 정리에도 그렇게 비용이 상당한가?
    > 
    > - 동작 변경 비용: 테스트, 검증, 배포 리스크
    > - 코드 정리 비용: 리뷰, 맥락 공유, CI, 충돌, 되돌리기, 검토 시간
    > 
    > 쿼리스트링을 nuqs로 변경한 사항도 해당되는 것일까
    > 
    > - 구현 방식만 바뀐 것 같아도 사용자가 체감할 수 있으면 동작 변경
    
- 일괄 처리 규모를 줄이면 검토 비용이 줄고 코드 정리 비용이 줄인다.
    
    > 작은 배치에도 고정비용 존재. 규모를 너무 줄이면 관리 비용이 증가
    
- 팀에 신뢰와 강력한 문화가 있으면 코드 정리 후에는 굳이 검토할 필요 없다. 검토하지 않더라도 코드 정리가 안정성을 해치지 않으면 상호작용 위험이 줄어든다.
    
    > 이런 경우는 테스트코드로 전부 방어할 수 있다는 말과 같나?
    > 
    > - 테스트만 많다고 자동으로 되는 것이 아니라 팀의 작업 방식 전체를 말함
    >     - 작은 변경을 만드는 습관 / 구조 변경과 동작 변경을 섞지 않는 규율 / 일관된 코드 스타일 / 리뷰어와 작성자 사이의 공감대 / 되돌리기 쉬운 배포 방식
    

## 19장 리듬

- 시스템 동작을 더 쉽게 변경하기 위해 코드 정리 진행
    
- 리듬을 관리한다 : 코드 정리를 관리하는 기술 중 하나
    
- 소프트웨어 설계는 프랙탈이라 설계는 크게도 작게도 가능하다.
    
    > 프랙탈? - 비슷한 패턴이 크기를 바꿔 반복된다.
    
- 코드 정리 시간을 분 단위를 유지하되, 한 시간을 넘기지 않도록한다. 한 시간 이상이 걸린다면, 동작 변경을 위해 필요한 최소한의 구조 변경 시기를 놓친 것이다.
    
- 코드가 너무 엉망이라서 몇 시간 동안 코드 정리 선행하는 것도 유리할 수도 있다.
    
- 동작 변경은 코드 안에서 뭉쳐서 나타나는 경향이 있다.
    
    > 핫스팟
    > 
    > - 실제 기능 변경은 자주 손대는 영역에 반복적으로 몰리는 경향있음
    
- 코드 정리를 많이 하게되면 나중에는 정리된 코드 안에서 동작 변경을 진행하게 된다.
    
- 대부분의 코드에 손을 대지않아도 정리되지 않은 코드를 만나는 일이 급격하게 줄어든다.
    

## 20장 얽힘 풀기

- 코들 정리하다보면 동작 변경사이에서 전환 시점을 놓치기도 한다.
- 변경 사이의 선후 문제는 보통 시간이 지나면 해결된다. (21장)

## 21장 코드 정리 시점

코드 정리하고 동작변경할지, 동작 변경하고 코드 정리할지 혼란스럽다!

- **아예 하지않기**
    - 코드의 동작이 앞으로 바뀌지 않을 믿음이 있다면 가능
    - 설계를 개선하더라도 배울 것이 없을 때
- **나중에 정리하기 : 의도적 미루기**
    - 정리할 코드 분량 많지만 보상이 보이지 않을 때
    - 내가 원할때 하면 더 의욕적이기도 하다~
- **동작 변경 후에 코드 정리**
    - 코드정리를 시간이 지나고 진행하면 맥락을 잊어버릴 수 있어서 정리비용이 증가한다.
    - 방금 고친 코드를 다시 변경 예정일 때 / 지금 정리하는 것이 더 저렴할 때 / 동작 변경 시간과 거의 비슷할 때
- **코드 정리 후에 동작 변경**
    - 작은 정리 하나만 해도 다음 동작 변경이 훨씬 안전하고 쉬워질 때, 먼저 정리한다.
    - 코드 정리 자체를 목적으로 삼지 않도록 경계해야한다.
